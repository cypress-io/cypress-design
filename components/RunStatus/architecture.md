## Source provenance

Ported from `cypress-io/cypress-services` — there's no single "RunStatus" component there. The visual is composed inline in:

- `frontend/packages/dashboard/src/runs/RunsListItem.tsx` — status icon + `#N` rendering, branch via `CommitBranch`
- `frontend/packages/dashboard/src/overview/RunAnalytics/Chart/RunDurationTooltip/index.tsx` (`BasicDetails`) — the `.run-build-number` pill with per-status border colors
- `frontend/packages/dashboard/src/overview/RunAnalytics/Chart/RunDurationTooltip/RunDurationTooltip.scss` (lines ~51–132) — `.run-build-number`, `.base-run`, `.past-run`, and the per-status border colors

The port:

- Drops application coupling (`@reach/router`, `RunStatusIcon` from `@frontend/design-system`, the `getDesignSystemStatus` mapper, the project/runs URL builders, `data-pendo` analytics).
- Replaces SCSS color variables with Tailwind utility classes pulled from existing cypress-design tokens (jade-400, red-400, indigo-400, gray-400, orange-400). No new color tokens introduced.
- Generalizes the inline "base run" vs "past run" CSS classes into a single `variant: 'base' | 'link'` prop.
- Adds a Vue parallel implementation.

## Render structure

Simplified render tree (React; Vue mirrors this):

```
<div data-cy="run-status">                        (root wrapper: inline-flex; ref, fallthrough attrs, className)
  <span>                                          (the pill: flex, items-center, rounded, ::after overlay)
    <Segment kind="buildNumber" />                (always rendered)
    {branch && <Segment kind="branch" />}         (rendered only when branch is set; preceded by ::after divider)
  </span>
</div>
```

The wrapper `<div>` is the root: `ref`, fallthrough attributes, and `className` land on it (DS convention). `pillClassName` lands on the inner pill `<span>` via `tailwind-merge`.

Each `Segment` is:

```
<a|span data-cy="run-status-{kind}">              (<a> when href is truthy, else <span>)
  <Icon />                                        (StatusIcon | technology-branch-h_x16)
  <span>{label}</span>                            (`#${buildNumber}` | branch name)
</a|span>
```

The build-number segment's icon is `StatusIcon`; the branch segment's icon is from `@cypress-design/{react,vue}-icon`.

## Status → icon mapping

| Status      | StatusIcon variant | StatusIcon size | Border color (`variant="link"`) |
| ----------- | ------------------ | --------------- | ------------------------------- |
| `passed`    | `solid`            | `16`            | `border-jade-400`               |
| `failed`    | `solid`            | `16`            | `border-red-400`                |
| `running`   | `outline`          | `16`            | `border-indigo-400`             |
| `cancelled` | `solid`            | `16`            | `border-gray-400`               |
| `errored`   | `solid`            | `16`            | `border-orange-400`             |
| `timedOut`  | `solid`            | `16`            | `border-orange-400`             |
| `noTests`   | `solid`            | `16`            | `border-orange-400`             |
| `overLimit` | `solid`            | `16`            | `border-orange-400`             |

`running` falls back to `outline` because that's the only variant `StatusIcon` defines for it — see `components/StatusIcon/constants/src/constants.ts`. All other statuses use `solid` (their richest variant).

Mapping table lives in `constants/src/index.ts` as `STATUS_VARIANTS` (status → `{ variant, size }`) and `STATUS_BORDER_CLASSES` (status → border-color class). The component reads both at render time. **Don't inline the colors at call sites — the table is the single source of truth and is shared between React, Vue, and the icon-registry `<tw-keep>` comments.**

## Constants keying

- **`CssClasses`** — flat object of static classes:
  - `container` — root wrapper (`inline-flex`, `pointer-events-auto`)
  - `pill` — the pill `<span>` (`flex items-center`, rounded, height, the `::after` overlay base classes)
  - `pillBaseVariant` — base-variant additions (subtle bg, no outer border)
  - `pillLinkVariant` — link-variant additions (border-width 1px, the per-status border color is appended separately)
  - `segment` — each segment base (padding, gap, items-center)
  - `link` — segment as `<a>` (no underline, hover/focus styles, `focus-visible:outline`)
  - `unlinked` — segment as `<span>`
  - `icon` — icon spacing
  - `branchText` — `max-w-[260px] truncate`
  - `divider` — `::after` separator applied to the first segment when the second segment is present (`after:content-[''] after:border-r after:border-gray-100 after:h-3 after:mx-1 after:self-center`)
- **`CssTheme`** — keyed by `'light' | 'dark'`, sub-keyed by element role:
  - `pill` — base background + text color for the pill
  - `link` — link text color + `hover:`/`focus-visible:` background, applied to each `<a>`
  - `divider` — `after:border-*` color, merged into the `divider` class on the leading segment
- **`STATUS_VARIANTS`** — `Record<RunStatusKey, { variant: VariantStatusIconProps['variant']; size: '16' }>` — used by both React and Vue to pick the icon.
- **`STATUS_BORDER_CLASSES`** — `Record<RunStatusKey, string>` — used only when `variant === 'link'`.

Types are derived from constants with `keyof typeof`:

- `type RunStatusTheme = keyof typeof CssTheme`
- `type RunStatusKey = keyof typeof STATUS_BORDER_CLASSES`

## Theme strategy

Single `CssTheme[theme]` lookup. No `dark:` modifier classes; the parent's `dark` class is **not** consulted. Matches the convention established by `RunResults`, `Textbox`, `Tag`, etc., and the "explicit theme prop" decision recorded in `implement-component.md`.

### Border is an `::after` overlay, not a real border

Same pattern as `RunResults` — see [its architecture doc](../RunResults/architecture.md) for the full rationale. Short version:

1. **Height.** A real `border` adds 2px to the box, regressing the pill from 24px to 26px. The overlay adds nothing to layout.
2. **Visible on hover.** Linked segments fill the box and paint their hover background over the pill's own background/border. A border, inset shadow, or inset outline on the pill all get covered. The `::after` is the last-painted positioned child, so it draws _above_ the segments and stays visible. `after:pointer-events-none` keeps it from intercepting clicks.
3. **Preflight-independent.** Renders correctly even where a consumer disables Tailwind preflight (e.g. cypress-services).

Border color comes from `STATUS_BORDER_CLASSES[status]` — applied to the `::after` overlay's `box-shadow`/`border` declaration via `tailwind-merge` when `variant === 'link'`. When `variant === 'base'`, no border is drawn.

## Segment divider

When `branch` is set, a 1px vertical divider renders between the build-number and branch segments. Implementation: the first segment gets a class — `CssClasses.divider` — that applies `after:content-[''] after:border-r after:border-gray-100 after:h-3 after:mx-1 after:self-center` (light theme), plus the dark divider class from `CssTheme.dark.divider`.

This mirrors `RunResults`'s separator pattern (between leading and regular stats). Don't refactor to a sibling element — assistive tech would announce it as a list item / separator-with-role, and the `data-cy` contract assumes the segments are the only direct children of the pill.

## Link rendering

Three-way decision per segment: no href → `<span>`; href + `renderLink` callback → caller-provided element (caller wraps in their router's link); href, no `renderLink` → native `<a href>`. See the `Segment` helper in `react/RunStatus.tsx` and `renderSegment` in `vue/RunStatus.vue` for the actual implementation.

`renderLink` is a unified **function prop** in both React and Vue — not a Vue scoped slot. Signature: `(href: string, children: unknown, className?: string) => unknown`. Same callback handles both `href` and `branchHref` — there's no need for two distinct callbacks because both segments share the same router and styling rules; the only difference between calls is the URL.

The `aria-label` and `data-cy="run-status-{kind}"` end up on the rendered element. The native `<a>` path applies both; the `renderLink` path delegates that responsibility to the caller (noted in `instructions.md`).

## Variant + linkability

`variant` and `href` are **independent**. Four combinations:

| `variant` | `href` set | Result                                                                     |
| --------- | ---------- | -------------------------------------------------------------------------- |
| `base`    | no         | Plain pill, no border. Pure presentational.                                |
| `base`    | yes        | Plain pill, `#N` segment is clickable. No status-colored border.           |
| `link`    | no         | Status-colored border, but `#N` segment is plain text. Rare but supported. |
| `link`    | yes        | Status-colored border + clickable `#N`. The "past run" treatment.          |

Same applies orthogonally to `branchHref` for the branch segment.

## State handling

CSS pseudo-classes only. No JS state. Focus styling lives in `CssClasses.link`; theme-dependent hover background/text lives in `CssTheme[theme].link` (no `dark:` modifiers):

- hover background: `hover:bg-gray-50` (light) / `hover:bg-gray-900` (dark), with `hover:no-underline` and theme-matched `hover:text-*`
- `focus-visible:outline focus-visible:outline-2 focus-visible:outline-indigo-500 focus-visible:outline-offset-0`

The pill itself does not get hover styling — only the `<a>` segments do. An unlinked pill (no `href`, no `branchHref`) is fully static.

## Forwarding and extension points

- React forwards `ref` to the root wrapper `<div>` (`HTMLDivElement`).
- React component is **not** `React.memo`-wrapped — consumers opt into memoization.
- Vue exposes the component via `defineComponent`; props are typed from the shared `RunStatusProps` interface.

## Gotchas

- The `running` status only supports `outline` in `StatusIcon` — don't try to force `solid` for it (you'll fall back silently and look wrong on inspection). The `STATUS_VARIANTS` table encodes this and is the source of truth.
- `cancelled` is the design-system spelling. Cypress-services internally uses `canceled` in some places — translate at the boundary, not in this component.
- The divider must **not** render when only one segment is present. Check `branch` before applying `CssClasses.divider`.
- `className` appends to the root wrapper via `clsx` (can't override DS classes — only add). To override a pill utility (e.g. `bg-*`, `border-*`), use `pillClassName`, which is merged onto the pill `<span>` via `tailwind-merge`.
- Don't add `dark:` modifier classes anywhere — theming is via explicit `theme` lookup.
- Status border classes must carry a `<tw-keep>` comment in `STATUS_BORDER_CLASSES` so Tailwind's tree-shaker keeps them. Without this, the dynamic `border-jade-400` / `border-red-400` / etc. get stripped from production builds.
- `data-cy` segment kinds are kebab-case (`run-status-build-number`, `run-status-branch`) — DOM convention. Internal segment kind union is camelCase (`'buildNumber' | 'branch'`). Convert at the boundary via a small inline helper; don't propagate kebab-case into the TS API.
