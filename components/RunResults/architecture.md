# RunResults — architecture

How the RunResults component is built. Read this when extending, fixing, or rebuilding the component. For usage, see [`instructions.md`](./instructions.md).

## Source provenance

Two-part port from `cypress-io/cypress-services`:

- **Test-counts pill** — ported from `frontend/packages/design-system/src/components/RunStats.tsx` + `module.RunStats.scss`. The original is React-only; the Vue version in this repo is a parallel implementation matching the React behavior. The port drops application coupling (router, query-string builder, `data-pendo` analytics, project/build URL construction) and remaps SCSS variables to Tailwind utility classes.
- **Run-status pill** — ported from the inline composition in `frontend/packages/dashboard/src/runs/RunsListItem.tsx` (status icon + `#N` rendering, branch via `CommitBranch.tsx`) and `frontend/packages/dashboard/src/overview/RunAnalytics/Chart/RunDurationTooltip/index.tsx` + `RunDurationTooltip.scss` lines ~51–132 (the `.run-build-number` pill with per-status border colors, `.base-run` and `.past-run` modifiers). The port:
  - Drops application coupling (`@reach/router`, `RunStatusIcon` from `@frontend/design-system`, the `getDesignSystemStatus` mapper, the project/runs URL builders, `data-pendo` analytics).
  - Replaces SCSS color variables with Tailwind utility classes pulled from existing cypress-design tokens (`jade-400`, `red-400`, `indigo-400`, `gray-400`, `orange-400`). No new color tokens introduced.
  - Generalizes the inline "base run" vs "past run" CSS classes into a single `runStatus.variant: 'base' | 'link'` prop.
  - Adds a Vue parallel implementation alongside React.

The original cypress-services treatment renders these as two separate, ad-hoc inline elements in the same row. Folding them into a single `RunResults` component matches how they're always used in practice (and was already anticipated by the existing wrapper `<div>`).

## Why no separate `FlakyBadge` component

The cypress-services source has a sibling `FlakyBadge` React component. We don't port it. In the new design, flaky is shown as a plain icon + count just like the other stats — no dedicated badge component needed inside RunResults. (A "flaky badge" visual exists in other parts of the Cypress UI, but it's unrelated to this component and lives in its own place.)

## Render structure

Simplified render tree (React; Vue mirrors this):

```
<div data-cy="run-results">            (root wrapper: inline-flex, gap, fallthrough attrs)
  {runStatus && (
    <span data-cy="run-status">        (run-status pill: flex, items-center, rounded, ::after border overlay)
      <Segment kind="buildNumber" />   (status icon + #N)
      {runStatus.branch && (
        <Segment kind="branch" />      (branch icon + branch text, preceded by ::after divider on the previous segment)
      )}
    </span>
  )}
  {hasAnyStat && (
    <ul>                               (test-counts pill: flex, items-center, rounded, ::after border)
      {/* Leading stats (independent, each optional) */}
      {flaky > 0 && <Stat status="flaky" />}
      {showSelfHealed && <Stat status="selfHealed" />}
      {/* Regular stats */}
      {(expanded || skipped > 0) && <Stat status="skipped" />}
      {(expanded || pending > 0) && <Stat status="pending" />}
      {(expanded || passed > 0) && <Stat status="passed" />}
      {(expanded || failed > 0) && <Stat status="failed" />}
    </ul>
  )}
</div>
```

The wrapper `<div>` is the root: `ref`, fallthrough attributes, and `className` land on it (DS convention). It holds up to two pills:

- The **run-status pill** is a `<span>` (not `<ul>` — its contents are segments, not a list). Rendered only when `runStatus` is provided.
- The **test-counts pill** is the existing `<ul>` of stats. Rendered only when at least one stat would be visible.

`pillClassName` (top-level) merges into the test-counts `<ul>`. `runStatus.pillClassName` merges into the run-status `<span>`. Both via `tailwind-merge`. Neither pill knows about the other — the wrapper handles the gap via `inline-flex gap-1` (or similar; finalize in stage 2).

Flaky and self-healed are conceptually independent — not a unit. They render in fixed left-to-right order when both are present, but neither implies the other.

### Early return when both pills are empty

Before the JSX above, compute:

- `hasRunStatus = !!runStatus` (just truthiness — the prop's mere presence renders the pill)
- `hasAnyStat = (flaky > 0) || showSelfHealed || (expanded || skipped > 0) || (expanded || pending > 0) || (expanded || passed > 0) || (expanded || failed > 0)`

If both are `false`, return `null` (React) / render nothing (Vue `v-if` at the template root). This is the documented behavior — see "Empty state" in `instructions.md`.

Each `Stat` is:

```
<li data-cy="total-{status}">
  <Tooltip overlay={…}>            (only when showTooltip)
    <a|span>                       (<a> when links?.[status] is a truthy href, else <span>)
      <Icon />                     (StatusIcon | IconStatusFlaky | IconGeneralSparkleSingleSmall)
      <span>{count}</span>
    </a|span>
  </Tooltip>
</li>
```

The Tooltip lives **inside** the `<li>`, wrapping the inner `<a>` (linked) or `<span>` (unlinked) — _not_ the `<li>` itself. The shared `@cypress-design/{react,vue}-tooltip` renders a `<div ref=...>` around its children, and inserting that `<div>` between `<ul>` and `<li>` would violate list semantics. The whole stat surface still triggers the tooltip because the inner `<a>` / `<span>` uses `w-full` plus the `<li>`-level padding, so hover/focus anywhere on the stat hits the Tooltip's reference element.

## Separator

A vertical divider renders between the leading stats (flaky and/or self-healed) and the regular stats. The source uses a `::after` pseudo-element on the last rendered leading `<li>` with a 1px right border, 12px tall, vertically centered.

Implementation: the last rendered leading `<li>` gets a class — `CssClasses.separatorAfter` — that applies `after:content-[''] after:border-r after:border-gray-100 after:h-3 after:mx-1 after:self-center` (light theme), plus the dark separator class from `CssTheme.dark`.

Note: source uses 2px horizontal margin on the separator (`margin-left: 2px; margin-right: 2px`). We round to 4px (`mx-1`) to satisfy the design system's multiple-of-4 spacing rule — record this as an intentional deviation, not a bug.

Which `<li>` gets `separatorAfter` is computed at render time:

- If `showSelfHealed`: self-healed is the last leading stat → it gets `separatorAfter`.
- Else if `flaky > 0`: flaky is the last leading stat → it gets `separatorAfter`.
- Else: no leading stats, no separator.

The separator must **also** be suppressed when no regular stats render (all four regular counts are zero and `expanded=false`). Compute "will any regular stat render?" first; only apply `separatorAfter` if yes. This keeps the separator from dangling at the end of the pill.

## Status → icon mapping

### Test-counts pill (per-stat)

| Stat                                        | Icon                                                        | Comes from                                                            |
| ------------------------------------------- | ----------------------------------------------------------- | --------------------------------------------------------------------- |
| `passed` / `failed` / `skipped` / `pending` | `StatusIcon` with a matching `status` prop                  | `@cypress-design/react-statusicon` + `@cypress-design/vue-statusicon` |
| `flaky`                                     | `IconStatusFlaky`                                           | `@cypress-design/react-icon` + `@cypress-design/vue-icon`             |
| `selfHealed`                                | `IconGeneralSparkleSingle` — native 12px (jade stroke/fill) | same as flaky                                                         |

See `renderIcon` in `vue/RunResults.vue` and the `icon` factory inside `Stat` in `react/RunResults.tsx` for the actual wiring (icon names → component, fixed `size="12"`, the flaky-yellow-bg workaround via the `iconFlaky` class).

### Run-status pill (per-status)

| `runStatus.status` | StatusIcon variant | StatusIcon size | Hover border (`variant="link"`) |
| ------------------ | ------------------ | --------------- | ------------------------------- |
| `passed`           | `solid`            | `16`            | `jade-400`                      |
| `failed`           | `solid`            | `16`            | `red-400`                       |
| `running`          | `outline`          | `16`            | `indigo-400`                    |
| `cancelled`        | `solid`            | `16`            | `gray-400`                      |
| `errored`          | `solid`            | `16`            | `orange-400`                    |
| `timedOut`         | `solid`            | `16`            | `orange-400`                    |
| `noTests`          | `solid`            | `16`            | `orange-400`                    |
| `overLimit`        | `solid`            | `16`            | `orange-400`                    |

`running` falls back to `outline` because that's the only variant `StatusIcon` defines for it — see `components/StatusIcon/constants/src/constants.ts`. All other statuses use `solid` (their richest variant). Branch icon is `technology-branch-h_x16` from the icon registry; size `16` to match the pill height.

The resting-state border on **both** variants is the neutral `gray-100` (light) / `gray-800` (dark) applied via `CssTheme[theme].runStatusPill`. The status color above only appears on `:hover` of a `variant="link"` pill (via `hover:after:shadow-[...]` in `RUN_STATUS_BORDER_CLASSES`).

Mapping tables live in `constants/src/index.ts`: `RUN_STATUS_VARIANTS` (status → `{ variant, size }`), `RUN_STATUS_TEXT_CLASSES` (status → build-number text color), and `RUN_STATUS_BORDER_CLASSES` (status → hover border class). The component reads all three at render time. **Don't inline the colors at call sites — the tables are the single source of truth and are shared between React, Vue, and the icon-registry `<tw-keep>` comments.**

**Runtime guard.** Both `RunStatusPill` (React) and `renderRunStatusPill` (Vue) look up `RUN_STATUS_VARIANTS[status]`; if the value is `undefined` (an un-mapped domain enum or partial mid-load data), they return `null` — the pill is skipped and the surrounding `RunResults` still renders. A `console.warn` fires in development. This keeps a data drift from crashing the whole tree.

## Constants keying

- **`CssClasses`** — flat object of static classes. Names use a `runStatus*` prefix for the new pill so they don't collide with the existing test-counts classes:
  - **Wrapper:** `container` (the root `<div>` — `inline-flex`, gap, `pointer-events-auto`).
  - **Test-counts pill** (existing): `list` (`<ul>` — `flex`, `items-center`, rounded, the `::after` overlay), `item` (each `<li>` base), `link` (the `<a>` inside a linked stat), `unlinked` (the `<span>` inside an unlinked stat), `icon` (default icon margin), `iconFlaky` (flaky icon override — `path:first-child` fill transparent), `iconSelfHealed` (self-healed icon margin), `separatorAfter` (separator on the last leading `<li>`). The count text `<span>` has no dedicated class.
  - **Run-status pill** (new): `runStatusPill` (the `<span>` — `flex`, `items-center`, rounded, the `::after` overlay base — the resting 1px gray border comes from `CssTheme[theme].runStatusPill`, applied to **both** variants), `runStatusSegment` (each segment base: padding, gap, items-center), `runStatusSegmentDividerAdjacent` (`!pr-0` — applied to the first segment when a divider is present, so the divider sits at the segment boundary), `runStatusLink` (segment as `<a>`: no underline, hover/focus styles, `focus-visible:outline`), `runStatusIcon` (icon spacing, applies to both status and branch icons), `runStatusBranchText` (`max-w-[260px] truncate font-normal`), `runStatusDivider` (`::after` divider applied to the first segment when the second segment is present).
- **`CssTheme`** — keyed by `'light' | 'dark'`, with each entry further sub-keyed by element role:

  - `list` — border + base text colors, applied to the test-counts `<ul>`
  - `link` — link text color + `hover:`/`focus-visible:` background, applied to each test-counts `<a>`
  - `separator` — `after:border-*` color, merged into the `separatorAfter` class on the leading `<li>`
  - `runStatusPill` — base bg + text colors for the run-status pill
  - `runStatusLink` — link text + hover bg for run-status segments
  - `runStatusDivider` — `after:border-*` color for the segment divider

- **`RUN_STATUS_VARIANTS`** — `Record<RunStatusKey, { variant: VariantStatusIconProps['variant']; size: '16' }>`. Used by both React and Vue to pick the StatusIcon variant/size.
- **`RUN_STATUS_BORDER_CLASSES`** — `Record<RunStatusKey, string>` of `hover:after:shadow-[...]` classes. Applied only when `variant === 'link'`, and only fires on hover (the pill's default gray border comes from `CssTheme[theme].runStatusPill`). Must carry `<tw-keep>` comments so Tailwind's tree-shaker keeps the dynamic hover-border colors.

Types are derived from constants with `keyof typeof`:

- `type RunResultsTheme = keyof typeof CssTheme`
- `type RunStatusKey = keyof typeof RUN_STATUS_BORDER_CLASSES`

  Don't apply `CssTheme[theme]` whole-cloth to a single element — pick the sub-key that matches what you're rendering.

Types are derived from constants with `keyof typeof`:

- `type RunResultsTheme = keyof typeof CssTheme`

Icon size is a fixed `'12'` for all icons — no constant, no prop.

## Packaging — one published package per framework

`@cypress-design/constants-runresults` is **private** (`"private": true`) and exists only to share class strings, types, and pure helpers between the React and Vue builds. It is **not** published. Consumers install a single package — `@cypress-design/react-runresults` or `@cypress-design/vue-runresults` — and never see the constants package.

To make that work, the constants are bundled into each component's dist:

- **JS** — the constants are listed in each component's `devDependencies` (not `dependencies`), so they are not externalized: React's rollup externalizes only `Object.keys(pkg.dependencies)`, and the Vue config omits constants from `extraExternal`. The bundler inlines the constants' JS.
- **Types** — re-exporting types `from '@cypress-design/constants-runresults'` would leak that specifier into the published `dist/index.d.ts`, pointing consumers at a package they don't install. So each package emits a **self-contained** declaration: React via `rollup-plugin-dts`, Vue via `vite-plugin-dts` (rollup types). The constants types are inlined; the real runtime deps (`react`, `clsx`, sibling `@cypress-design/*`) stay referenced by package name. Verify after a build: `dist/index.d.ts` must contain no `constants-runresults` references.

The sibling `vue-statusicon` / `vue-tooltip` (and the React equivalents) stay externalized — they are separately published packages with their own consumers.

## Tooltip integration

### Placement

Per-stat: `top-start` for `flaky`, `top-end` for every other stat. The cypress-services source used `placement={isFlaky ? 'top' : 'topRight'}`; we deviate on flaky and use `top-start` so the popper is left-aligned with the stat and the arrow points at the stat itself rather than at the center of a wider popper. Non-flaky stats keep `top-end` (cypress-services' `topRight` maps cleanly to Floating UI's `top-end`, since our internal Tooltip uses Floating UI placement names).

### Color follows the inverse of `theme`

`theme="light"` → Tooltip rendered with `color="dark"`. `theme="dark"` → Tooltip rendered with `color="light"`. Cypress-services source did not theme the tooltip at all; we invert deliberately so the tooltip contrasts with whatever surface the pill itself is matching.

### Labels

Built by a helper `getTooltipLabel(status, count, isLinked)` in `constants/src/index.ts`. Pure function, no React/Vue imports.

`isLinked` is computed in the component layer as `!!links?.[statKey]` — i.e. the consumer set a truthy href for this stat. It is **not** inferred from the rendered output (so passing a `renderLink` that returns plain text still resolves to `isLinked: true`; the truthy value of `links?.[statKey]` is the source of truth). An empty-string href is falsy and resolves to `isLinked: false`.

The `links` prop is optional. Vue defaults it to `{}`; React reads it with optional chaining. Either way, omitting `links` entirely — or omitting any individual key — is safe and resolves to `isLinked: false` for that stat.

Returns:

- `flaky`, count `n`:
  - `n === 1`: `"This test both passed and failed when retried within a run"`
  - else: `"{n} tests both passed and failed when retried within a run"`
- Other statuses, linked (`isLinked: true`): `"View {status} tests"`
- Other statuses, unlinked (`isLinked: false`): `"{Capitalized status} tests"`

Inline ternary for the flaky singular/plural — no `pluralize` dependency. The source used `pluralize('test', value)` but for a single call site a ternary is clearer and avoids a tree-shake-resistant dep.

`selfHealed` is treated like any other status for label purposes (`"View self-healed tests"` / `"Self-healed tests"`). When rendering "self-healed" inside the label text or a `data-cy` attribute, the helper converts the camelCase key to kebab-case via a small inline helper — single call site, no util dep.

The Vue template can call the helper as `getTooltipLabel(...)`; React calls it directly. Avoid re-implementing the strings in both files.

## Theme strategy

The cypress-services source's `darkTheme` boolean only adjusts the separator color. We extend it to a full themed palette (border, text, hover background, separator) because:

1. The pill is meant to live on dark surfaces, where the default light gray-100 border becomes invisible.
2. The repo convention is `theme: 'light' | 'dark'` (see Textbox, Tag, TestResult — to be cross-checked in Stage 2).

Approach: a single `CssTheme[theme]` lookup. No `dark:` modifier classes; the parent's `dark` class is **not** consulted. This matches Textbox's strategy and is consistent with the "explicit theme prop" decision recorded in `implement-component.md`.

### Border is an `::after` overlay, not a real border

The 1px pill border is drawn as an absolutely-positioned `::after` overlay (an inset box-shadow on the pseudo-element, color from `CssTheme[theme].list`), not a `border`/`outline`/inset-shadow on the `<ul>` itself. Three reasons:

1. **Height.** A real `border` adds 2px to the box, regressing the pill from 24px to 26px. The overlay adds nothing to layout.
2. **Visible on hover.** Linked stats fill the box and paint their hover background over the `<ul>`'s own background/border. A border, inset shadow, or inset outline on the `<ul>` all get covered. The `::after` is the last-painted positioned child, so it draws _above_ the stats and stays visible. `after:pointer-events-none` keeps it from intercepting link clicks.
3. **Preflight-independent.** Renders correctly even where a consumer disables Tailwind preflight (e.g. cypress-services).

### Class-name props: `className`, `pillClassName`, `runStatus.pillClassName`

Three escape hatches, each scoped to a specific element:

- `className` — root wrapper `<div>`. Appended via `clsx` (DS convention — Button/Textbox do the same). Can only **add** classes, not override.
- `pillClassName` — test-counts pill `<ul>`. Merged via `tailwind-merge` so conflicting utilities (`bg-*`, `border-*`) override the DS default:

  ```ts
  twMerge(CssClasses.list, CssTheme[theme].list, pillClassName)
  ```

- `runStatus.pillClassName` — run-status pill `<span>`. Same `tailwind-merge` treatment, scoped to that pill only.

`tailwind-merge` resolves conflicting utilities by keeping the **last** one, so `pillClassName="bg-gray-900"` wins over the theme's `bg-gray-1000` (and `bg-transparent` works) — without the Tailwind source-order problem a plain append (`clsx`) would hit, where two `bg-*` utilities leave the winner up to stylesheet order. Non-conflicting classes (the `::after` border shadow, text size) are untouched.

Why two pill escape hatches (rather than one): the two pills carry independent visual treatments (different borders, different backgrounds when bridging from a colored surface), so collapsing them into one prop would force consumers to write either both or neither. Keeping them separate matches the API split between `runStatus` and the test-counts props.

The Vue side passes the wrapper class array straight to Vue (which normalizes string/array/object fallthrough) and runs each `pillClassName` through `clsx` + `twMerge` on its respective pill element.

## State handling

CSS pseudo-classes only. No JS state. Focus styling lives in `CssClasses.link`; the theme-dependent hover background/text lives in `CssTheme[theme].link` (no `dark:` modifiers — see Theme strategy):

- hover background: `hover:bg-gray-50` (light) / `hover:bg-gray-900` (dark), with `hover:no-underline` and theme-matched `hover:text-*`
- `focus-visible:outline focus-visible:outline-2 focus-visible:outline-indigo-500 focus-visible:outline-offset-0`

The `<li>` itself does not get hover styling — only the `<a>` does. This matches the source: an unlinked stat shouldn't change appearance on hover.

## Link rendering

Same pattern in both pills. Three-way decision per linkable element:

- No href → `<span>` (plain text).
- href + `renderLink` callback → caller-provided element (caller wraps in their router's link).
- href, no `renderLink` → native `<a href>`.

See the `Stat` helper in `react/RunResults.tsx` and `renderStat` / `renderSegment` in `vue/RunResults.vue` for the actual implementation.

`renderLink` is a unified **function prop** in both React and Vue — not a Vue scoped slot. Signature: `(href: string, children: unknown, className?: string) => unknown`. **One callback for both pills** — it's called separately for each linkable element with the matching href. `children` is typed `unknown` because the two frameworks pass different shapes: in React it's a JSX element (icon + count, or icon + label), in Vue it's an **array** of `VNode`s. `className` is the element's computed link styling (gray text, `px-[6px]`, no-underline, hover/focus) — the caller applies it to their router link so it matches the default `<a>`. The class strings differ slightly between the test-counts and run-status pills (different padding / icon spacing), so the caller should always apply the `className` received in that specific call, not a global one.

For the test-counts pill, the `aria-label` and `data-cy="link-{status}"` end up on the rendered element. For the run-status pill's build-number segment, `aria-label="View run #{buildNumber}"` and `data-cy="run-status-build-number"`. The branch segment is always a plain `<span>` (never a link) with `data-cy="run-status-branch"` and no `aria-label`. The native `<a>` path applies aria-label + data-cy; the `renderLink` path delegates aria-label responsibility to the caller (noted in `instructions.md`).

## Self-healed gating

A single condition: `showSelfHealed === true`. When the consumer sets it, the stat renders regardless of `selfHealed` count — including `0`. The count is displayed verbatim (`null` is coerced to `0` like the regular stats).

The consumer decides visibility based on whether the run could have self-healed tests at all (e.g. `cy.prompt` was available). When that's not the case, the consumer sets `showSelfHealed: false` and the stat is hidden entirely. The component does not second-guess the flag based on count.

`expanded` does **not** affect leading stats — it's a regular-stats-only knob. Flaky still only renders when its count is `> 0`; self-healed is governed entirely by `showSelfHealed`.

## Forwarding and extension points

- React forwards `ref` to the root wrapper `<div>` (`HTMLDivElement`).
- React component name should not be `memo`-wrapped at the file boundary — let consumers opt into memoization. (Source is `React.memo`-wrapped; we drop that to keep the API ergonomic for `forwardRef` and ref attachment.)
- Vue exposes the component via `defineComponent`; props are typed from the shared `RunResultsProps` interface.

## Gotchas

- The flaky `IconStatusFlaky` has a path-fill quirk (`fill: transparent !important` on the first path) in the source SCSS. The icon component may handle this internally now — verify in Stage 2; if not, the workaround is a wrapper class on the `<li>` that targets `svg path:first-child`.
- The separator inside the test-counts pill uses `::after` on the last leading `<li>`, not a separate sibling element. Don't refactor to a sibling `<li role="separator">` — it changes the semantics (assistive tech announces it as a list item) and breaks the existing `data-cy` selector contract. Same applies to the run-status pill's segment divider — `::after` on the build-number segment, not a sibling.
- The run-status pill's divider must **not** render when only one segment is present. Check `runStatus.branch` before applying the divider class.
- `running` is the only run-status that doesn't support `solid` in `StatusIcon` — don't try to force it. The `RUN_STATUS_VARIANTS` table encodes this and is the source of truth.
- `cancelled` is the design-system spelling. Cypress-services internally uses `canceled` (American) in CSS; translate at the boundary (`runStatusEnumToStatusDesignSystem`), not in this component.
- `null` counts are coerced to `0` for both display and "is this stat empty" logic. Don't introduce a difference between `null` and `0` without a strong reason.
- Don't add `dark:` modifier classes anywhere — theming is via explicit `theme` lookup.
- `className` appends to the root wrapper via `clsx` (can't override DS classes — only add). To override a pill utility (e.g. `bg-*`), use `pillClassName` (test-counts) or `runStatus.pillClassName` (run-status), which are merged via `tailwind-merge` — those are the documented escape hatches.
- `RUN_STATUS_BORDER_CLASSES` entries must carry `<tw-keep>` comments. Without them, Tailwind's tree-shaker strips the dynamic `border-jade-400` / `border-red-400` / etc. from production builds.
- `StatKey` is camelCase (`selfHealed`) matching repo convention (see `StatusIcon`'s `noTests`, `timedOut`, `overLimit`). `data-cy` attributes use kebab-case (`status-icon-self-healed`, `total-self-healed`, `run-status-build-number`) — DOM convention. Convert at the boundary via a small inline helper; don't propagate kebab-case into the TS API.
